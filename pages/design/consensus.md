# 共识层

跨社团的关键决策(调度、治理、积分)需要不可篡改的多方一致记录。JLUCraft 用 **Raft** 作共识协议——比 PBFT 简单,适合 3-7 节点。

## 共识组设计

- **共识组成员**:各社团社长指定的服务器节点(通常是各社团性能最好的那台)。
- **共识组规模**:3 / 5 / 7 节点(奇数),允许 `(n-1)/2` 节点故障。
- **共识范围**:
  - 实例调度决策
  - 节点加入 / 退出批准
  - 治理操作(多签指令执行记录)
  - 预言机积分发放
  - 节点信誉分更新

## 节点信誉系统 (Node Score)

| 维度 | 权重 | 含义 |
| ---- | ---- | ---- |
| uptime_score | 0.4 | 过去 30 天在线率(0-100) |
| performance_score | 0.4 | 服务质量和延迟(0-100) |
| governance_score | 0.2 | 参与共识投票率(0-100) |
| penalty | — | 恶意行为扣分,直接减 |

最终分数 = 各维度加权求和后减去惩罚分。

### 信誉分影响

- **调度权重**:高分节点获得更多实例
- **中继优先级**:高分节点优先被选为中继
- **治理投票权**:低于阈值则暂时失去投票权

## Raft 在 JLUCraft 中的实现要点

- `consensus` 节点完整参与 Raft 日志复制和选举
- `worker` 节点仅观察,不参与投票
- 每 10000 条日志生成快照,存储到 S3
- 新节点从 S3 拉取快照 + 增量日志快速同步
- 共识日志条目可通过哈希追溯,任何节点可以独立验证

::: tip 为什么选 Raft 而不是 BFT
社团 3-7 节点,持有共识节点的社长是硬编码公钥持有者,假定不会主动作恶。拜占庭行为的成本由[治理层](./governance.md)多签和[预言机](./oracle.md)反作弊机制处理。
:::
