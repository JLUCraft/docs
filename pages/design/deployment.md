# 部署拓扑

以下是一个**典型 JLUCraft 社团部署**的参考拓扑，可作为新节点接入的模板。

## 基础组件清单

| 组件 | 数量 | 说明 |
| ---- | ---- | ---- |
| 引导节点 (Bootstrap) | 1 ~ N | 公网 VPS，仅作 DHT 入口 |
| 共识节点 (Consensus) | 至少 1 | 加入 Raft 组，参与调度与治理 |
| 工作节点 (Worker) | 任意 | 仅运行实例，不参与共识 |
| S3 存储 | 1 套 | 持久化层（AWS S3 / R2 / MinIO 自建均可） |
| TURN fallback | 0 ~ 1 | 极端情况下的中转 |

## 多节点参考部署

```mermaid
flowchart TB
  subgraph Shared["共享基础设施"]
    BS[公网引导节点]
    S3[(S3 存储)]
    TURN[TURN fallback]
  end

  subgraph A["节点组 A（主服务器）"]
    A1["共识节点<br/>(云服务器)"]
    A2["高性能 PC<br/>(内网穿透)"]
    A3["N× 玩家客户端"]
  end

  subgraph B["节点组 B（校园网）"]
    B1["共识节点<br/>(云服务器)"]
    B2["服务器 ×2<br/>(校园网)"]
    B3["N× 玩家客户端"]
  end

  subgraph C["节点组 C（备用）"]
    C1["共识节点<br/>(云服务器)"]
    C2["服务器<br/>(家中)"]
    C3["N× 玩家客户端"]
  end

  subgraph Raft["共识组（3 节点 Raft · 容忍 1 节点故障）"]
    A1 --- B1 --- C1
  end

  A2 --> S3
  B2 --> S3
  C2 --> S3
  A3 -.-> BS
  B3 -.-> BS
  C3 -.-> BS
```

## 节点角色与位置选型

| 节点角色 | 推荐位置 | 关键考量 |
| -------- | -------- | -------- |
| 引导节点 | 公网 VPS，廉价小内存即可 | 必须有公网 IP，带宽足够支撑 DHT 查询 |
| 共识节点 | 社团云服务器 | 在线率 ≥ 99%、低延迟、稳定 |
| 工作节点 | 校园网 / 宿舍 PC | 性能优先，内网穿透由网络层处理 |
| TURN fallback | 公网 VPS | 仅打洞失败时启用，带宽消耗大 |

## 网络与穿透

详细策略见 [网络层 — NAT 穿透策略](./network.md#nat-穿透策略)，拓扑层面确保：

- **每个节点组至少有一台公网节点**：用于自身内网节点对外打洞，也可担任其他节点组的中继备选。
- **引导节点冗余 ≥ 2**：防止单点宕机导致新节点无法接入。
- **S3 存储跨区域备份**：唯一可靠持久化层，本身需要高可用。

## 扩展接入流程

新节点加入网络的步骤：

```mermaid
flowchart TB
  S1[部署共识节点 + 工作节点] --> S2[生成节点 PeerID]
  S2 --> S3[提交给现有共识组]
  S3 --> S4[现任管理成员发起 add_node 多签]
  S4 -->|达到阈值| S5[写入共识层]
  S5 --> S6[新节点连接引导节点同步快照]
  S6 --> S7[加入 Raft 组]
```

::: warning 新管理成员公钥
若新管理成员公钥需注册到 genesis，要全体核心成员同意并发布新二进制（物理边界，见 [安全边界](./security.md)）。
:::
